## Headline findings (read these first)

1. **P0 — Statutory citations are unverified.** Two of the six agents either invented a BIF Act section or accepted a wrong section the user typed without correction. A real lawyer would not ship the output as-is.
   - Payment Schedules: agent emitted "served in accordance with section 18 of the Building Industry Fairness (Security of Payment) Act 2017 (Qld)" when asked for the relevant section for service of a schedule. Real section is **76**. The user did not supply "18" — the agent invented it.
   - General Correspondence: when the user prompted with "section 80 of the BIF Act" for a suspension notice, the agent accepted it verbatim. Right to suspend is **section 98** under BIF 2017 (Qld). Section 80 is the consequence-of-no-schedule provision, not suspension.
2. **P0 — Agent does not push back on legally wrong premises.** Delay Costs agent will happily draft a delay-costs claim for an abnormal-rainfall delay event under AS 4902, even though abnormal weather is a qualifying cause for **EOT only** under that contract — it is not compensable for delay costs absent a contract Schedule modification. The agent should refuse, or at least flag, before committing. It did not.
3. **P0 — Agent invents numbers in tables.** Payment Schedules agent put `$312,400` in the "Claimed" column of the disputed-items table when no claimed amount was supplied in the prompt and the figure cannot be derived from the inputs given. The number happens to coincide with the total of VAR-001 in the project library, suggesting context bleed.
4. **P1 — Wrong AS 4902 clause.** EOTs agent cited "clause 35.5 of the Contract" for the reservation-of-rights paragraph at the end of an EOT notice. AS 4902 EOT lives at clause 34 (specifically 34.3 / 34.5 / 34.7). The agent took the only clause number it had seen in the prompt (cl 35.5 was the *abnormal-rainfall threshold* in the Schedule) and reused it as the EOT-giving clause.
5. **P1 — Em dash in every drafting-agent page-sub.** All six agents have an em dash in the descriptive subtitle directly below the page title. Ted's typography rule says no em dashes in user-visible copy. The em dashes are in `AGENT_DESCRIPTIONS` (`site/assets/sopal-v2/sopal-v2.js`, around lines 57–66 in the live JS).
6. **P1 — Hardcoded chat-empty-state hint copy is wrong on five of six agents.** The hint above the chat composer reads: *Try "Set the claimed amount to $487,250", "Add a section reserving rights to delay costs", or "Tighten the language on the statutory endorsement".* This is Payment-Claims-specific. It surfaces unchanged on Payment Schedules, EOTs, Variations, Delay Costs, and General Correspondence.

The bottom-line assessment is at the end of this document.

## Per-agent findings

### 1. Payment Claims — `NEEDS FIX`

**What worked.**
- Starter template loads cleanly with the expected sections (claimant, respondent, project, reference date, date served, amount, identification, breakdown table, statutory endorsement, service, signature).
- Statutory endorsement cites `section 75 of the Building Industry Fairness (Security of Payment) Act 2017 (Qld)` — that is the correct section for making a payment claim.
- AI substitution via the right-pane chat works. Asked it to substitute placeholders with project parties + `$487,250` claimed + `31 March 2026` reference date + `7 April 2026` service date, and it cleanly populated every header field, the claimed amount, the identification narrative, the service paragraph, and the signature line. It correctly left line-item descriptions and the signatory's personal name as `[Item description]` / `[Signatory name]` because no facts were supplied. Good behaviour.
- Document HTML is clean — no raw tags leaking into the rendered editor; tables and lists round-trip through the contenteditable.

**Bugs.**
- **P1 — Em dash in page-sub.** "Review or draft payment claim material — BIF Act compliance, work identification, dates, service, evidence." Source: `AGENT_DESCRIPTIONS["payment-claims"]`. **Fix:** replace the em dash with a colon or a full stop.
- **P1 — Chat composer scrolls off-screen on long documents.** Once the draft fills the editor, the chat composer (textarea + send button + project-context checkbox) is laid out at the bottom of the right column and is pushed below the fold. The user has to scroll the whole page to access it. There is no sticky-bottom on `.drafting-chat-form` inside `.drafting-chat-card`. Confirmed via DOM rect: textarea at `y = 1006`, viewport height `775`. **Fix:** make `.drafting-chat-form` sticky to the bottom of the right pane and let `.drafting-chat-stream` scroll inside it.
- **P2 — Hardcoded chat-empty-state hint copy** (see headline finding 6 above).
- **P2 — Starter template does not auto-prefill known project metadata.** Claimant, respondent, project name, contract reference are sitting in `project.claimant / respondent / name / reference` but the template renders them as `[Claimant name]` etc. The user has to ask the AI to substitute them. A first-pass prefill at template-load would cut friction. **Fix:** in `getProjectDraft` (around live line 5320), interpolate project metadata into the template before storing.

### 2. Payment Schedules — `BROKEN`

**What worked.**
- Starter template structure is right (issuer, claimant, payment claim being responded to, scheduled amount, reasons, table, reservation, signature).
- AI substitutes party names, project, and the scheduled amount correctly. Reservation-of-rights paragraph is good standard wording.

**Bugs.**
- **P0 — Hallucinated BIF section.** Output ended with: *"This payment schedule is served in accordance with section 18 of the Building Industry Fairness (Security of Payment) Act 2017 (Qld)."* The user prompt asked it to "cite the relevant BIF section for service of payment schedules" and supplied no number. Section 18 is not in the SOPA chapter at all. The right section is **76**. This is the kind of error that loses a respondent the schedule on a jurisdictional argument. **Fix:** the system prompt for the drafting agents needs a hard rule that BIF section numbers must come from a small allowlist (64, 67, 68, 73, 75, 76, 79, 82(4), 83, 85, 88, 98, 99) and any other number must be flagged as `[verify section]` instead of emitted. Or feed the agent a small statutory crib-sheet in the system prompt for QLD work.
- **P0 — Invented claimed amount in table.** The disputed-items table emitted `Claimed: $312,400 / Scheduled: $290,000`. The user supplied only the scheduled amount and the disputed component values ($112,250 + $85,000); the agent never asked for, and was never given, a claimed total. `$312,400` matches the total in the project's `VAR-001 Latent rock cost summary` library doc — i.e. the agent appears to have reached into the project context and grabbed an unrelated number to fill an empty cell. **Fix:** when the claimed amount cannot be derived from the prompt, emit `$[Claimed]` rather than substituting a context-bleed number. Add an explicit instruction in `EDIT_DRAFT_SYSTEM_PROMPT` that empty cells must remain placeholders and must not be filled from the project context unless that specific fact is in the prompt.
- **P1 — Reasons-for-difference paragraph echoes the user's prompt verbatim.** I supplied two reasons in compressed form (`(a) variation VAR-001 latent rock $112,250 disputed pending instruction; (b) liquidated damages $85,000 deducted for 17 days of delay`). The agent dropped that string into the body unchanged rather than expanding it into proper schedule prose with the contractual basis cited per item. The whole point of the agent is the rewrite. **Fix:** strengthen the system prompt to require expansion of compressed user input into discrete, properly framed reasons.
- **P1 — Em dash in page-sub.**
- **P2 — Hardcoded hint copy.**

### 5. Delay Costs — `BROKEN`

**What worked.**
- Starter template loads. Header fields, the 12-day compensable period, the Aconex programme reference, and the three-row quantum table (site preliminaries / project management staff / tower crane standing charge) were all populated correctly. Quantum maths checks out (`$58,200 + $26,400 + $14,160 = $98,760`). Riverstone signs.

**Bugs.**
- **P0 — Agent does not refuse a legally unsound prompt.** The user prompt asked it to draft a delay-costs claim for a 12-working-day delay caused by abnormal rainfall. Under unmodified AS 4902, abnormal weather is a qualifying cause for **EOT only**; it is **not** a compensable delay event for delay costs unless the contract Schedule reverses the default position. The agent did not flag this, did not ask whether the Schedule modifies the default, did not push back. It just produced the document. A lawyer using this output to advise the client would be in trouble. **Fix:** for delay-costs-specific drafting, the system prompt needs an explicit gate — "before drafting, identify whether the cause of delay is *compensable* under the contract; if it is not, refuse to draft and state the problem". The same gate should apply when reviewing a received delay-costs claim.

Either commit to a clause number (34.3 / 34.5 / 34.7 as applicable) or leave a `[clause #]` placeholder. Mushy prose in the entitlement section is a tell that the underlying analysis is mushy.

## Overall assessment

These agents are not yet usable on a real matter. The UI is solid — Word-style editor, clean chat-driven rewrites, predictable templates, sensible save state, no broken HTML. As a layout and interaction model, this is the right shape and a real user would not get confused. The problem is the legal substance underneath. Across six agents I caught one invented BIF section, one wrongly accepted BIF section, one wrong AS 4902 clause, one invented dollar figure pulled from unrelated project context, and one drafting agent that will happily write a delay-costs claim for a non-compensable event without flagging it. The common cause is that the system prompt for the drafting agents has no statutory crib-sheet, no contract-form crib-sheet, and no rule that when a fact is missing the agent must emit a placeholder rather than a guess. Everything I caught is fixable in `EDIT_DRAFT_SYSTEM_PROMPT` plus a small statutory/contractual reference block — the editor and template plumbing do not need to change. Until that is done, the safest framing for early users is "this is a *first-draft generator*, every section must be cross-checked against the actual Act and the actual contract before serving" — and that warning needs to be louder than it is today.
